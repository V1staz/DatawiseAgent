## 2026-04-17 第 1 轮清障

### 1. 默认运行路径切换到无 Docker 本地执行

**问题**

- 原仓库默认 `use_docker: True`。
- 当前协作复现路线明确采用 API-first + 无 Docker。
- 若维持原默认配置，仅填写 `OPENAI_API_KEY` 与 `OPENAI_BASE_URL` 后无法直接启动会话执行链路。

**修改**

- 修改 `configs/default_config.yaml`
  - `code_executor.use_docker: False`
  - `code_executor.use_proxy: False`
- 修改 `configs/datamodeling.yaml`
  - 改为本地执行默认配置
  - `use_docker: False`
  - `use_proxy: False`
  - `use_gpu: False`
  - 保留 `image_name` 仅作兼容字段
- 新增：
  - `configs/docker_default_config.yaml`
  - `configs/docker_datamodeling.yaml`
  用于保留原始容器化路线
- 修改 `.env.example`
  - 明确共享 fork 默认使用本地 no-Docker 配置
  - 补充切回 Docker 配置的说明

**原因**

- 共享仓库当前的主要目标是降低启动门槛，使“填 key/base_url 即可启动”成为默认路径。
- 同时保留 Docker 配置文件，便于后续做高保真对齐实验。

**验证**

- 读取配置文件确认默认值已切换为 `use_docker: False`
- `main.py`、`DatawiseAgent`、`LocalJupyterServer` 成功导入

### 2. 为 Docker 失败场景补充自动回退

**问题**

- 即使后续某些 session 仍传入 `use_docker=True`，在本机未配置 Docker 或 Docker 不可用时，当前代码会直接失败。

**修改**

- 修改 `datawiseagent/agents/datawise_agent.py`
- 在 `create_new_session()` 中加入：
  - Docker executor 创建失败时记录 warning
  - 自动回退到 `LocalJupyterServer`
  - 同步将当前 session 的 `use_docker` 状态改为 `False`

**原因**

- 共享复现环境需要更强的容错性。
- 当前路线已明确以无 Docker 为主，自动回退符合使用预期。

**验证**

- 代码通过 `py_compile`
- 本地创建 session 时 `session.code_executor_config.use_docker` 为 `False`

---

### 3. 去除评测脚本中的硬断点

**问题**

以下脚本存在 `pdb.set_trace()`，会导致批量复现卡住：

- `evaluation/eval_data_modeling.py`
- `evaluation/MatplotBench/scripts/model_eval.py`
- `evaluation/InfiAgentBench/scripts/eval_closed_form.py`
- `evaluation/DataModeling/scripts/show_results.py`

**修改**

- 删除或绕开上述脚本中的调试断点
- 对 `show_results.py` 保留异常抛出，避免静默失败

**原因**

- 共享复现需要无人值守执行能力
- 断点残留会破坏批跑流程与评测自动化

**验证**

- 目标脚本通过 `py_compile`
- repo 中剩余的 `pdb.set_trace()` 仅存在于：
  - 注释/文档文本
  - `__main__` 测试块或注释掉的历史代码

### 4. 修复 MatplotBench 评分脚本的 API 读取方式

**问题**

- `evaluation/MatplotBench/scripts/model_eval.py` 仅从本地文本文件读取 key/url
- 无法直接复用 `.env` 中的 `OPENAI_API_KEY` / `OPENAI_BASE_URL`
- 同时脚本内还保留调试断点

**修改**

- 评分脚本改为优先读取：
  - `OPENAI_API_KEY`
  - `OPENAI_BASE_URL`
- 若环境变量不存在，再回退到：
  - `evaluation/MatplotBench/scripts/api_key.txt`
  - `evaluation/MatplotBench/scripts/url.txt`
- 删除断点

**原因**

- 与共享仓库的 API-first 路线统一
- 减少额外手工配置入口

**验证**

- 脚本通过 `py_compile`
- 代码路径中不再强依赖本地文本文件作为唯一入口

### 5. 修复视觉工具的硬编码配置

**问题**

- `datawiseagent/tools/dsbench/vision_tool.py` 中 `evaluate_image()` 仍写死：
  - `api_key = "xxx"`
  - `base_url = "xxx"`
- 导致带 visual tool 的 MatplotBench 路径无法直接复用统一环境变量

- 删除过时异常分支中的 1.x 兼容风险写法

**原因**

- 使视觉工具与主 API 配置统一
- 允许后续按需单独指定视觉模型 API

**验证**

- 模块通过 `py_compile`
- 模块成功导入

### 6. 修复无 planning 分支中的 `self_debug` 真值问题

**问题**

- `datawiseagent/agents/datawise_agent.py` 中原实现：

```python
is_self_debug = (session.agent_config.debug.self_debug,)
```

- 因多余逗号形成非空 tuple，逻辑上恒为真
- 会污染后续 ablation 与配置语义

**修改**

- 改为真正的布尔值：

```python
is_self_debug = session.agent_config.debug.self_debug
```

**原因**

- 保证配置开关语义与运行行为一致

**验证**

- 代码通过 `py_compile`

### 7. 调整 DataModeling 初始化环境检查，移除自动安装 torch

**问题**

- `chat_history_initialization(..., tool_mode="datamodeling")` 中原逻辑会在 session 初始化时直接执行：
  - `pip install ... torch`
- 这与当前“无 Docker + 不装 torch + API-first”路线冲突
- 同时每次会话初始化自动改环境会增加不稳定性

**修改**

- 修改 `datawiseagent/agents/datawise_agent.py`
- DataModeling 初始化改为：
  - 仅检查系统信息
  - 仅展示已安装核心库版本
  - 以 Python 方式检测 `torch` / `transformers` 是否存在
  - 不再自动执行 `pip install torch`
- 默认模式初始化也不再在每个 session 启动时执行通用包安装

**原因**

- 本地环境 `ds` 已预装常用分析依赖
- 自动装包会污染共享环境并延长启动时间
- 当前路线优先保障可控和稳定

**验证**

- `tool_mode="default"` 创建 session 成功
- `tool_mode="datamodeling"` 创建 session 成功
- 初始化 cell 数量正常生成
- datamodeling 初始化输出中能够显示 `torch` / `transformers` 是否安装，而不会自动安装

### 9. 增加 DashScope-compatible 模型设置与环境变量入口

**问题**

- 共享复现需要支持不同兼容 OpenAI 协议的 provider。
- 当前协作成员使用的是 DashScope compatible mode。
- 需要区分：
  - 主 agent 使用的文本/代码模型
  - 视觉工具与可视化评测使用的视觉模型

**修改**

- 新增 `configs/setting_model.yaml`
  - 默认主模型设为 `qwen2.5-72b-instruct`
  - 在文件中保留 `qwen2.5-32b-instruct` 作为可切换选项
- 修改 `datawiseagent/tools/dsbench/vision_tool.py`
  - 支持从 `VISION_TOOL_MODEL` 读取视觉模型名
- 修改 `evaluation/MatplotBench/scripts/model_eval.py`
  - 支持从 `MATPLOT_JUDGE_MODEL` 读取评测视觉模型名
- 在 WSL 用户环境与项目 `.env` 中写入：
  - `DASHSCOPE_API_KEY`
  - `OPENAI_API_KEY`
  - `OPENAI_BASE_URL`
  - `CUSTOM_CONFIG=configs/setting_model.yaml`
  - `VISION_TOOL_API_KEY`
  - `VISION_TOOL_BASE_URL`
  - `VISION_TOOL_MODEL=qwen2.5-vl-72b-instruct`
  - `MATPLOT_JUDGE_MODEL=qwen2.5-vl-72b-instruct`

**原因**

- 主 agent 的 notebook 推理链路以文本和代码为主，默认应使用非 VL instruct 模型。
- 图像理解与可视化评测链路需要视觉模型，因此单独使用 `qwen2.5-vl-72b-instruct` 更合适。
- 把 provider 配置写入 `.env` 与 WSL 环境变量后，协作同学只需要替换 key/base_url 或模型名即可切换 provider。

**验证**

- 通过 `.env` 读取配置后，直接调用 DashScope compatible API：
  - 模型：`qwen2.5-72b-instruct`
  - 返回：`SMOKE_TEST_OK`
